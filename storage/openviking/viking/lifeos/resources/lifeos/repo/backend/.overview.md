# backend

This directory serves as the root container for the LifeOS backend application, a FastAPI-based system designed to manage personal life, spirituality, and productivity workflows. It encompasses the core application source code, dependency management, containerization configurations, and standalone utility scripts for testing and maintenance. The structure facilitates a production-ready environment using Python 3.12, Docker, and asynchronous programming patterns. This directory is suitable for backend developers, DevOps engineers, and system architects interested in Python web services, database connectivity, and agent-based service architectures.

## Detailed Description

### Dockerfile Dockerfile
This Dockerfile defines a containerized Python environment for running the backend application. It starts from the lightweight `python:3.12-slim` base image and sets the working directory to `/app`. The script installs `curl`, creates a non-root user named `appuser` for security, and copies the requirements file to install dependencies. It then copies the application source code from the `app/` directory, sets appropriate file ownership, and switches to the non-root user. The container exposes port 8000 and uses `uvicorn` to launch a FastAPI application, allowing configurable host and port settings through environment variables.

### clear_sandbox.py clear_sandbox.py
This Python script serves as a standalone utility to reset the memory state for a specific sandbox agent environment. The code defines a single asynchronous entry point, the `main` function, which executes the memory clearing logic. It imports the `clear_memory` function from the `app.services.memory` module, indicating a dependency on the application's internal service layer for data persistence management. The script utilizes Python's `asyncio` library to handle asynchronous execution, running the event loop via `asyncio.run(main())` when executed directly. In the broader project architecture, this file acts as a maintenance or operational tool, allowing developers to manually purge conversation history or stored context for the "sandbox" agent without running the main application.

### test_db.py test_db.py
This file serves as an asynchronous Python script for testing database connectivity and querying specific Agent records. The primary component is the `main` coroutine, which utilizes `asyncio` to manage an asynchronous database session via `async_session` imported from `app.database`. It constructs a SQLAlchemy `select` query targeting the `Agent` model, filtering for an entry where the name equals 'sandbox'. The script executes this query, retrieves a single result using `scalar_one_or_none`, and prints the agent's name and JSON configuration if found, otherwise indicating the agent was not found. As a standalone test utility, it validates the Object-Relational Mapping (ORM) configuration and database connection logic, playing a supportive role in the backend architecture by ensuring data retrieval workflows function correctly before integration into the main application.
